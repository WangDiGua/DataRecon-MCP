你观察得非常敏锐。在处理 AI 与数据库直接交互的任务时，**“只读账号”和“简单正则拦截”确实只是“安全防盗门”的最外层**。如果真的投入生产环境，这个计划确实存在几个足以导致系统崩溃或数据泄露的“隐蔽炸弹”。

我们必须从**“防御性编程”**的角度，将安全层级从单纯的“禁止写操作”提升到“全链路资源与逻辑隔离”。

---

## 隐藏的安全风险与深度防御方案

### 1. 语义级的“逻辑炸弹”：拒绝服务攻击 (DoS)
**风险**：模型虽然执行的是 `SELECT`，但如果它编写了一个**笛卡尔积查询**（如 `SELECT * FROM big_table_a, big_table_b`）或者多重嵌套的复杂 `JOIN`，即便只返回 200 行，数据库的 CPU 和内存也会在执行瞬间被拉满，导致整个库宕机。

**防御补丁：**
* **执行时间硬上限**：在 Node.js 执行前，在 SQL 层面注入超时指令。例如针对 MySQL：`SELECT /*+ MAX_EXECUTION_TIME(5000) */ ...`（限制执行时间为 5 秒）。
* **SQL 复杂度预审**：使用 **AST（抽象语法树）解析器**（如 `shorthash` 或 `sql-parser`）分析 SQL 的 `JOIN` 数量和嵌套深度。超过 3 层 `JOIN` 直接拒绝。



---

### 2. 绕过正则的 SQL 注入
**风险**：模型可能被“诱导”或因错误生成带有特殊编码、注释符（如 `--` 或 `/* */`）的 SQL，从而绕过简单的字符串匹配，执行多条语句。

**防御补丁：**
* **禁用多语句执行**：在 `mysql2` 配置中，严格设置 `multipleStatements: false`。
* **强制 AST 校验**：不再仅仅检查“是否有 DELETE 关键字”，而是通过 AST 解析确认该 SQL 的 **Statement Type** 必须且只能是 `SelectStatement`。如果解析出两个 Statement，直接拦截。

---

### 3. 内存缓冲区引发的 OOM (内存溢出)
**风险**：我们讨论过“临时内存”，如果子 Agent 疯狂调用 `store_to_memory`，或者单行数据包含极大的 `TEXT/BLOB` 字段，Node.js 进程会迅速耗尽内存。

**防御补丁：**
* **双重配额管理**：
    * **行数配额**：单 Session 不超过 5000 行。
    * **字节配额**：单 Session 总大小限制在 10MB 以内。
* **流式处理**：在写入内存前，通过 `Buffer.byteLength` 预估数据量，一旦超标立刻触发 `MemoryPressureError`。

---

### 4. 权限溢出：敏感库/表泄露
**风险**：`list_datasources` 可能会列出 `mysql` 系统库、`performance_schema` 或者包含敏感薪资、密钥的业务库。模型不应该知道这些库的存在。

**防御补丁：**
* **动态黑名单（Catalog Filtering）**：
    * 在 `list_datasources` 中硬编码拦截系统库名。
    * 在 `get_table_list` 中增加过滤，禁止访问包含 `user`, `config`, `secret`, `salary` 等关键词的表。
* **元数据掩码**：即便模型查到了列名，如果字段名命中了敏感词，在 `get_table_schema` 阶段就对字段进行脱敏处理。

---

### 5. 侧信道与错误信息泄露
**风险**：原始 MySQL 错误（如 `ER_ACCESS_DENIED_ERROR`）可能包含内网 IP、库名路径或版本号。攻击者可以利用这些信息进行渗透。

**防御补丁：**
* **错误分类器**：在返回给模型之前，先对错误进行过滤。
    * **业务错误（如字段名错）**：原样透传，方便 AI 纠错。
    * **系统错误（如连接超时、权限拒绝）**：转化为通用的“Database System Error”，隐藏底层设施信息。

---

## 优化后的安全架构图 (Defense in Depth)



### 完善后的全量开发清单 (Security Check)

| 风险维度 | 防御技术 | 实现位置 |
| :--- | :--- | :--- |
| **执行过久** | `MAX_EXECUTION_TIME` 注入 | SQL 预处理层 |
| **逻辑越权** | AST 语法分析 (Only SELECT) | Node.js 逻辑层 |
| **数据过大** | LRU-Cache + 字节配额 | Memory 缓冲区 |
| **连接泄漏** | 自动销毁 idle 连接的连接池 | Database 驱动层 |
| **协议破坏** | 捕获所有 `console.log`，强制重定向到 `stderr` | 全局拦截 |
| **权限泄露** | `information_schema` 访问受限账号 | 数据库配置层 |

### 你的直觉是对的
单纯的代码逻辑无法完全防御 AI 的“幻觉”或“恶意诱导”。我们需要在 **数据库本身的权限隔离**、**Node.js 运行时的资源限制** 以及 **SQL 解析器的静态分析** 这三者之间建立联防机制。
